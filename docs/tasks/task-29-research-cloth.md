# Task 29（可选）：确定性布料系统调研

## 目的

为后续布料能力形成可评审的技术决策和实施任务草案。该任务只做调研与架构设计，不修改当前刚体引擎、不加入占位 API，也不把布料列入当前 product-ready 验收范围。

## 依赖

依赖 `setup.md` 与 Task 22A 冻结的 Gravity `jobs.Dispatcher` contract，但不
阻塞刚体主链。必须理解当前Q32.32、刚体约束行、资产、快照和C ABI边界；
不得引入第二套executor/scheduler，也不得假设当前主链会为布料牺牲正确性。

## 交付物

- `docs/research/future/cloth.md`；
- XPBD、隐式/显式 FEM、Projective Dynamics 的决策矩阵；
- 推荐方案的数据模型、求解流程、碰撞/自碰撞方案和刚体耦合边界；
- 固定点误差与范围预算、确定性并行策略、快照/回滚成本模型；
- 对 asset format、world phase、事件、查询、C ABI 的兼容性影响；
- dependency-closed、可独立执行的后续实现任务DAG和完成门禁；任务数量由真实
  ownership/validation边界决定，不为满足配额机械拆分；
- 至少 5 个原始论文或官方技术资料的链接、结论与适用限制。

## 详细实现架构

候选架构必须明确粒子、三角拓扑、逆质量、厚度、材料、stretch/shear/bend/area 约束以及 constraint lambda 的所有权。分析布料资产离线烘焙、运行时实例状态、tear-disabled 不可变拓扑，以及未来撕裂扩展时的版本边界。

碰撞章节必须覆盖 cloth–rigid 全 shape、布料边/点/面与 TriangleMesh/HeightField、连续碰撞候选生成、自碰撞 broadphase、邻接过滤、稳定 feature ID 和确定接触排序。不能用“复用现有碰撞”代替对粒子/边/三角 primitive 的具体映射。

求解章节必须比较布料约束与刚体 PGS 的单向、交替、统一 island 三种耦合方式，说明动量反馈、迭代顺序、warm start、睡眠和高速运动。并行方案必须定义无原子浮点累加的 graph coloring 或固定分区/稳定归并方式，并证明 worker 数变化不改变结果。

数值章节必须先验证Q32.32对长度、compliance、lambda、速度和校正量是否足够，
给出量纲与上界及`i128`或更宽中间值位置；若不足，允许结论推荐隔离的确定性
定点表示或尺度域，但不得把平台float/libm引入现有刚体状态。快照章节区分
资产、未来相关状态、可重建派生结构，并估算典型10k/100k粒子内存和带宽。

## 实施步骤

1. 从原始论文/官方资料提取三类方法的积分、收敛、刚度、固定步长和并行属性。
2. 建立与 Gravity 冻结约束一致的评分维度，逐项给出证据，选择一个推荐方案及一个被拒方案。
3. 画出资产、实例、约束批、碰撞代理、缓存和快照的数据流与所有权。
4. 为 cloth–rigid/self-collision 写出确定的候选生成、排序、流形/约束生成和求解顺序。
5. 做固定点范围推导和最坏内存/时间复杂度估算；列出需要通过离线数值实验验证的假设。
6. 定义 C ABI 的未来扩展方式，只描述新增 handle/descriptor/function，不预占当前 ABI 符号。
7. 拆成可落地实现任务；每项包含输入、输出、测试语料、性能预算与失败条件。

## 验证

- 每个核心结论可追溯到来源或完整推导；
- 决策矩阵覆盖稳定性、可控刚度、碰撞、自碰撞、确定性、并行、回滚、内存和 ABI；
- 推荐架构能解释 2D cloth surface 在 3D 世界中的完整运动，而不是退化成当前 2D preset；
- 明确当前核心需要保留的扩展点以及不应提前加入的 API；
- 后续任务足以由 Agent 只读该调研、`setup.md` 和各自任务后实现，不存在“自行选算法”等未决项；
- 文档不包含 mock、占位实现、跳过验证或把研究结论冒充产品能力的表述。

## 完成判定

调研通过架构评审，推荐算法、刚体耦合、固定点预算、确定性并行、碰撞/自碰撞、状态边界和任务拆分均有唯一可执行结论。只列候选、缺少来源或把关键选择留给实现者均不算完成。
