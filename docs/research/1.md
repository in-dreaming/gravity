# 一致性物理引擎技术调研

> 文档定位：这是产品决策前的通用技术调研，保留 2D、第三方引擎和分阶段路线作为行业方案比较，不是 Gravity 的实现规范。其中“建议先做 2D”等通用建议已被后续产品决策否决。Gravity 已冻结为一次完整交付 product-ready 的 3D Zig 引擎；唯一实施上下文见 [`../tasks/setup.md`](../tasks/setup.md)，架构见 [`../design/deterministic-physics-engine.md`](../design/deterministic-physics-engine.md)。
>
> 本文中的“一致性”指 **确定性（determinism）**：相同版本的模拟程序从相同初始状态出发，按相同顺序接收相同输入后，在每个逻辑 Tick 产生完全相同的可观察状态。本文重点讨论实时游戏中的刚体物理、帧同步与回滚；布料、流体和高精度工程仿真不作为第一阶段目标。

## 结论先行

一致性物理引擎并不是“把浮点数换成定点数”这么简单，而是一套端到端的确定性模拟系统。它至少包含五层契约：

1. **时间一致**：固定 Tick、固定子步数、固定求解迭代数；渲染帧率不能进入模拟。
2. **数值一致**：规定每种运算的精度、舍入、溢出、超越函数和编译选项。
3. **执行一致**：实体、碰撞对、接触点、约束、事件和并行归并都有稳定顺序。
4. **状态一致**：可回滚状态必须覆盖物理世界的全部未来相关状态，而不只是位姿。
5. **输入与资源一致**：创建/删除顺序、随机数、配置、碰撞网格和用户回调也必须确定。

对于工程落地，建议先明确所需的确定性等级：

| 等级 | 保证范围 | 典型用途 | 代价 |
|---|---|---|---|
| L1 同二进制复现 | 同一可执行文件、同硬件环境重复运行一致 | 回放、调试、自动测试 | 低 |
| L2 跨线程一致 | 工作线程数量和调度不同仍一致 | 多核服务器、客户端性能档位 | 中 |
| L3 跨平台逐位一致 | x86/ARM、Windows/Linux/macOS/WASM 结果一致 | P2P 帧同步、跨端回滚 | 高 |
| L4 回滚一致 | 保存状态、恢复后重演与未回滚路径一致 | Rollback Netcode | 很高 |

这四者不能互相替代。以 Box2D 3.1 为例，它声明支持 64 位平台的跨平台、跨线程确定性，但 FAQ 明确说明其不提供 rollback determinism；这意味着“正常向前跑一致”不自动等于“任意快照恢复后仍一致”[^box2d-sim][^box2d-faq]。

**推荐决策：**

- 若目标是快速交付 2D、无需引擎内部快照回滚，可优先验证 **Box2D 3.1+**。
- 若技术栈是 Rust/WASM，或需要现成的世界快照，优先验证 **Rapier deterministic 构建**；确定性模式会限制并行和 SIMD，需要实测吞吐[^rapier-det][^rapier-start]。
- 若目标是通用 3D、C++、可接受严格浮点约束，可优先验证 **Jolt Physics 的 `CROSS_PLATFORM_DETERMINISTIC`**；官方文档给出的参考性能损失约为 8%，但应以本项目场景复测[^jolt]。
- 若要求长期跨平台、跨编译器、跨版本可验证，并且物理规模可控，采用 **定点数/软件浮点 + 自有状态与回滚层** 的风险最可控，但开发量最大。
- 如果游戏对碰撞响应的需求有限，优先做“确定性运动学 + 查询/触发器”，不要一开始就自研完整 3D 刚体引擎。

## 1. 确定性的严格定义与边界

### 1.1 正确的判定方式

设完整模拟状态为 $S_t$，第 $t$ 个 Tick 的输入为 $I_t$，资源与配置为 $A$，状态转移函数为 $F$：

$$S_{t+1}=F(S_t,I_t,A)$$

确定性要求任何参与节点上的 $F$ 都给出同一个结果。工程上通常使用规范化序列化后的哈希验证：

$$H_t=Hash(SerializeCanonical(S_t))$$

这里的“相同”最好定义为 **逐位相同**，而不是“位置误差小于某个 epsilon”。近似相同只能判断视觉接近，不能保证接下来的碰撞分支、睡眠判断和事件顺序相同。

### 1.2 必须纳入状态的内容

仅保存刚体位置和速度通常不足以支持回滚。完整快照至少应评估以下内容：

- Tick 编号、确定性随机数状态；
- 刚体位置、旋转、线速度、角速度、力/力矩累积器；
- 激活/睡眠状态及计时器；
- 碰撞器、关节、稳定 ID、句柄代数（generation）；
- 广相结构，或能以唯一方式重建广相结构的规范化数据；
- 接触流形、接触点特征 ID、上一帧冲量（warm-start cache）；
- 岛结构或其确定性重建所需信息；
- CCD/TOI 中间状态；
- 运行时可变材质、过滤规则和游戏逻辑状态。

Jolt 的状态记录器可以保存/恢复物理状态并进入 validation mode 检查重演一致性，这类接口可以作为自研快照层的参考[^jolt-recorder]。

### 1.3 不在确定性核心中的内容

渲染插值、粒子特效、音频播放、相机震动可以是非确定性的，但它们不能反向影响模拟。它们属于“确认后提交”的副作用：预测帧可以暂存，帧确认后再播放；发生回滚时丢弃未确认副作用，避免重复播放音效或重复发奖励。

## 2. 刚体物理引擎需要实现什么

一个可用于游戏的最小刚体引擎可拆成下面的流水线：

```text
输入命令
  → 创建/删除与力的提交
  → 外力与速度积分
  → 广相碰撞检测
  → 窄相碰撞检测与接触流形
  → 岛划分
  → 接触/关节约束求解
  → 位置与旋转积分
  → CCD 修正
  → 睡眠、事件、状态哈希
```

### 2.1 数学与基础设施

需要实现或封装：

- 标量、`Vec2/Vec3`、矩阵、四元数、变换、AABB；
- 点积、叉积、归一化、平方根、倒数、三角函数；
- 规定零长度归一化、除零、NaN/Inf、溢出和饱和行为；
- 稳定实体 ID、稳定容器、确定性排序、确定性 RNG；
- 规范化序列化、快照、哈希与状态差异工具。

如果采用定点数，乘法必须使用更宽的中间类型。例如 Q32.32 的乘法原则上需要 128 位中间积后右移 32 位；除法需要先左移并明确除零与溢出策略。还需要自研或固化 `sqrt/sin/cos/atan2` 的查表或整数算法。Photon Quantum 使用 Q48.16，并明确禁止在模拟中从 `float` 转换，因为转换本身可能引入平台差异；其文档也展示了定点数的范围、精度和溢出权衡[^quantum-fp]。

### 2.2 刚体状态与积分

3D 刚体的核心状态通常包括：质心位置 $x$、姿态四元数 $q$、线速度 $v$、角速度 $\omega$、逆质量 $m^{-1}$、局部逆惯量 $I_{local}^{-1}$。世界空间逆惯量为：

$$I_{world}^{-1}=R I_{local}^{-1}R^T$$

实时引擎常用半隐式欧拉：

$$v_{t+1}=v_t+\Delta t\,m^{-1}F_t$$

$$x_{t+1}=x_t+\Delta t\,v_{t+1}$$

$$\omega_{t+1}=\omega_t+\Delta t\,I_{world}^{-1}(\tau_t-\omega_t\times I_{world}\omega_t)$$

姿态通过角速度积分后必须按确定规则归一化。固定 Tick 解决的是“各端积分次数相同”，不能单独解决浮点、遍历顺序或输入差异。Box2D 推荐固定时间步，并以 60 Hz、4 个子步作为常见配置示例，但具体值必须按场景性能和稳定性测试决定[^box2d-sim]。

### 2.3 碰撞形状与质量属性

第一阶段建议只支持：

- 2D：圆、胶囊、凸多边形、线段/静态链；
- 3D：球、胶囊、盒、凸包、静态三角网格；
- 复合形状可由多个基础形状组合；
- 动态凹体暂不支持，离线分解为凸体。

质量、质心和惯量必须由确定性烘焙工具生成，或用固定算法在所有端重建。凸包构建、网格焊接、三角形排序若依赖第三方浮点工具，输出应在构建期固化并进入资源版本哈希，不能让客户端各自烘焙。

### 2.4 广相碰撞检测

广相的任务是从 $O(n^2)$ 物体对中筛出可能重叠的候选对。常见结构：

- **Sweep and Prune（SAP）**：适合物体移动相对平稳的场景；端点相等时必须用稳定 ID 破平局。
- **动态 AABB Tree / BVH**：通用性强，插入、移动和查询通常为对数级；旋转/重建策略必须确定。
- **Uniform Grid / Spatial Hash**：适合尺寸接近且分布均匀的物体；哈希桶遍历不能直接决定求解顺序。

广相输出必须规范化为 `(minBodyId, maxBodyId)`，去重后按稳定键排序。不要把线程竞争产生的 append 顺序传给窄相或求解器。Box2D 3 的做法值得参考：各工作线程写本地 bitset，主线程用按位 OR 合并，从而不让线程完成先后影响全局顺序[^box2d-det]。

### 2.5 窄相检测与接触流形

窄相回答“是否碰撞、法线是什么、有哪些接触点”。常见组合：

- 球/胶囊/盒等常见对使用解析算法；
- 凸多边形/盒使用 SAT；
- 一般凸体使用 GJK 做相交/距离查询，穿透后用 EPA 或基于面裁剪生成流形；
- 凸体对静态网格先查 BVH，再逐三角形测试。

不能只输出一个最深点。稳定堆叠需要 2D 的接触点组或 3D 的接触流形，并在相邻 Tick 间用 **特征 ID** 匹配旧接触点，以复用上一帧冲量。所有退化情形都要规定破平局规则，例如：

- 多个分离轴深度相等时选哪一个；
- GJK 方向接近零时选哪条备用轴；
- EPA 多个最近面相等时按哪个面 ID；
- 接触点裁剪、删减和排序规则；
- 比较使用严格大小、量化阈值还是 epsilon。

这些分支往往比主体算法更容易造成失同步。

### 2.6 约束求解：Sequential Impulse / PGS

主流实时刚体引擎通常把接触和关节统一成速度约束。对约束 Jacobian $J$，相对约束速度为：

$$Jv+b=0$$

其中 $b$ 可包含穿透修正（Baumgarte bias）、恢复系数等。单条约束的有效质量：

$$K=J M^{-1}J^T$$

本次冲量增量：

$$\Delta\lambda=-\frac{Jv+b}{K}$$

然后更新广义速度：

$$v \leftarrow v+M^{-1}J^T\Delta\lambda$$

法向接触是单边约束，累计冲量需满足 $\lambda_n\ge0$；摩擦冲量近似满足：

$$|\lambda_t|\le \mu\lambda_n$$

Sequential Impulse 本质上是逐约束的 Projected Gauss-Seidel。它高效、易 warm start，但**顺序敏感**：接触点和关节的排列不同，有限迭代后的结果也不同。因此必须固定：

- 岛顺序、岛内刚体顺序；
- 约束类型批次顺序；
- 接触对、流形和接触点顺序；
- 速度/位置迭代数；
- warm-start 的读取和累计规则。

第一阶段建议支持距离、固定、转动/铰链和滑动关节；电机、限位、弹簧可以后加。过大的质量比、很硬的关节链和大时间步都会降低迭代求解器收敛质量，需要设定产品级限制。

### 2.7 PBD/XPBD 的正确位置

PBD/XPBD 是约束求解方案，不是确定性方案。它们仍然受数值格式、约束遍历顺序、并行归并和输入一致性影响。

PBD 直接投影位置约束 $C(x)=0$。XPBD 引入柔度 $\alpha$ 和累计拉格朗日乘子，使材料刚度较少依赖时间步与迭代次数：

$$\Delta\lambda=\frac{-C(x)-\tilde\alpha\lambda}{\nabla C M^{-1}\nabla C^T+\tilde\alpha},\qquad
\tilde\alpha=\frac{\alpha}{\Delta t^2}$$

$$\Delta x=M^{-1}\nabla C^T\Delta\lambda$$

XPBD 很适合布料、绳、软体和部分统一约束系统；经典 Sequential Impulse/PGS 仍是刚体接触的成熟选择。若首要目标是确定性刚体，不应仅为了“一致性”切换到 XPBD。XPBD 原论文和 Small Steps 论文适合作为后续软体与子步策略参考[^xpbd][^smallsteps]。

### 2.8 CCD、睡眠和事件

- **CCD**：对高速物体进行 shape cast / conservative advancement，求 Time of Impact。TOI 相等、迭代未收敛和最大次数退出都需确定规则。
- **睡眠**：依据线/角速度阈值及持续时间整岛睡眠。阈值附近的微小差异会改变是否继续参与模拟，因此睡眠状态必须进快照；早期可关闭睡眠降低调试复杂度。
- **事件**：Begin/Stay/End、Trigger、JointBreak 等事件必须按稳定键输出。回滚预测帧的事件不能直接产生不可撤销副作用。

## 3. 数值一致性的三条技术路线

### 3.1 原生浮点 + 受控工具链

做法是继续使用 IEEE 754 `float/double`，但锁定编译器语义和数学函数：

- 禁用 `fast-math`、重结合和不受控 FMA contraction；
- 固定舍入模式和 subnormal/FTZ/DAZ 行为；
- 避免平台 C/C++ 标准库的 `sin/cos/atan2/pow`，使用自有实现；
- 固定 SIMD 路径和指令集，或验证不同 SIMD 路径逐位相同；
- 禁止 NaN/Inf 进入模拟，明确 `-0` 规范化策略；
- 锁定编译器、版本、宏、优化和依赖版本，并纳入握手协议。

MSVC 文档明确指出 FMA 与分开的乘加会因为中间舍入不同而得到不同结果，`/fp:fast` 也允许重排和代数变换[^msvc-fp]。Jolt 的跨平台模式要求相同源码与宏，并建议 Clang 使用 `-ffp-model=precise -ffp-contract=off`、避免标准三角函数及 `std::sort`[^jolt]。Box2D 3 则通过 precise math、禁用 contraction 和自有三角函数实现 64 位平台跨平台确定性[^box2d-sim]。

优点是性能好、可复用成熟算法；缺点是工具链约束多，升级必须跑完整跨平台确定性 CI。

### 3.2 整数定点数

定点数用整数表示 $raw/2^F$。加减快且语义清晰，适合要求强可移植性的模拟，但它并非“免费确定”：

- 有限范围导致溢出，乘除需要更宽中间类型；
- 小量会量化为零，大/小尺度混用会显著损失精度；
- `sqrt`、归一化、四元数和三角函数需专门实现；
- 3D 碰撞中的多层点积/叉积更容易耗尽有效位；
- C/C++ 有符号溢出属于未定义行为，必须使用无符号运算、检查或饱和语义；
- 算法中的 epsilon、容差和世界尺寸都要重新标定。

因此要先做范围预算。例如根据最大世界坐标、最大速度、最大质量、最大冲量和最深表达式推导整数位与小数位，而不是默认选择 Q16.16。Photon Quantum 选择 Q48.16 是一种偏性能的工程取舍，不是通用答案[^quantum-fp]。

### 3.3 软件浮点 / 确定性数学库

软件浮点显式实现 IEEE 运算和舍入，可获得比定点更自然的动态范围，并隔离硬件差异；代价通常是更高 CPU 成本和更复杂的 SIMD 优化。它适合规模中等但范围跨度较大、又必须跨平台逐位一致的场景。

### 3.4 WebAssembly 不是自动确定性开关

WASM 能缩小不同原生编译器和 CPU 指令路径的差异，但仍不能替代完整契约：

- 输入、容器遍历、线程、资源和宿主回调仍可能不确定；
- 标准 `Math.sin/cos` 等宿主函数不能假设跨平台逐位一致；
- WebAssembly 规范允许浮点 NaN 的符号等细节存在非确定性[^wasm-spec]；
- 引擎版本、构建 feature 和快照格式仍需锁定。

Rapier 官方文档要求启用 `enhanced-determinism`、使用其确定性数学函数，并保持创建/删除顺序相同；其 Rust 确定性 feature 不能与 parallel/SIMD feature 同时启用[^rapier-det][^rapier-start]。因此“纯 WASM + Rust 天然一致”是不成立的。

## 4. 执行顺序与并行确定性

### 4.1 稳定 ID 是整个系统的主键

每个刚体、碰撞器、关节和接触对都应有不复用或带 generation 的稳定 ID。所有会影响模拟的排序都以 ID 作为最终破平局键，不能依赖：

- 指针/内存地址；
- 哈希表迭代顺序；
- 线程完成先后；
- 平台 `sort` 对等价元素的未规定顺序；
- 对象池空闲链表因历史操作产生的偶然次序。

创建与删除必须经由按 Tick 排序的命令缓冲区提交，例如排序键 `(tick, commandType, issuerId, sequence)`。Jolt 和 Rapier 都明确要求修改 API、创建和删除以相同顺序发生[^jolt][^rapier-det]。

### 4.2 可行的确定性并行模式

完全关闭多线程最容易保证一致，但不是唯一选择。可以采用：

1. 单线程先生成并排序稳定工作列表；
2. 每个 worker 只写固定索引区间或线程本地缓冲；
3. 归并按预先定义的 key，而不是按完成时间；
4. 避免浮点原子累加；并行 reduction 使用固定树形；
5. 不允许多个线程竞争创建全局对象/句柄；
6. 不同岛可以并行，但岛内求解顺序固定。

Box2D 3 展示了跨线程数一致的实现思路，但用户回调和上层游戏逻辑仍可能破坏它[^box2d-det][^box2d-sim]。

## 5. 与帧同步和回滚网络的结合

### 5.1 延迟式 Lockstep

所有节点收齐 Tick $t$ 的输入后才计算 $t$。网络只传输入，带宽低，结构简单；缺点是输入延迟受最慢玩家和网络抖动影响。

需要实现：

- Tick 时钟和输入缓冲；
- 输入序号、确认、重传或冗余发送；
- 玩家缺失输入策略；
- 周期状态哈希与失同步处理；
- 加入/重连时的权威快照。

### 5.2 Rollback

本地输入立即执行，远端输入未到时使用预测值。迟到输入与预测不同时：

1. 找到最早错误 Tick；
2. 恢复该 Tick 之前的快照；
3. 写入真实输入；
4. 快速重演到当前 Tick；
5. 渲染层平滑展示修正后的状态。

GGPO 将这种模式描述为输入预测与 speculative execution，以避免传统输入延迟[^ggpo]。真正难点不在网络包，而在模拟必须可快速保存、恢复和多次重演。

### 5.3 回滚层需要实现的组件

- `InputFrame[tick][player]`：真实/预测标志；
- 快照环形缓冲：全量快照 + 可选增量压缩；
- `SaveState/LoadState/AdvanceOneTick` 三个纯模拟接口；
- 规范化状态哈希和分层哈希；
- 最大预测窗口、最大回滚深度；
- 副作用日志与确认提交；
- 追帧预算，防止一次深回滚造成长卡顿；
- 权威节点校验、防作弊和失同步恢复。

内存粗略估算：若快照为 512 KiB、保留 120 Tick，全量快照约需 60 MiB/世界。优化顺序应是先保证正确，再评估结构化快照、页级 copy-on-write、delta、压缩或降低窗口。

### 5.4 状态哈希不能只哈希 Transform

Transform 相同但 warm-start 冲量、睡眠计时器或随机数状态不同，未来仍会分叉。建议：

- 每 Tick 计算轻量 64/128 位哈希；
- 每 N Tick 计算完整规范化快照哈希；
- 分层记录 world/body/contact/joint/gameplay hash；
- 首次不一致时二分 Tick，再逐字段 diff；
- 哈希只用于诊断/一致性检测时可选高速非密码哈希；涉及对抗性输入时使用带密钥 MAC 或密码学哈希。

## 6. 推荐的软件架构

```text
Presentation（非确定）
  渲染插值 / 音频 / 特效 / UI
              ↑ 已确认事件
Netcode & Rollback
  输入预测 / 快照环 / 重演 / 校验 / 重连
              ↓ TickInput
Deterministic Simulation
  游戏规则 / ECS / RNG / 物理世界 / 事件日志
              ↓ 只读资源句柄
Baked Assets
  碰撞几何 / 质量属性 / 材质 / 配置 / 版本哈希
```

核心 API 建议保持极小：

```text
CreateWorld(config, bakedAssets) -> World
SubmitCommands(world, tick, orderedCommands)
Step(world, tickInput) -> OrderedEvents
SaveSnapshot(world) -> CanonicalBytes
LoadSnapshot(world, bytes)
HashState(world) -> StateHash
DiffState(snapshotA, snapshotB) -> StructuredDiff
```

关键原则：模拟层不能读取墙上时钟、渲染 delta、系统随机源、文件系统、网络到达时间、locale 或未版本化的外部数据。

## 7. 自研最小可用版本（MVP）范围

### 7.1 建议先做 2D

若没有必须自研 3D 的产品约束，先做 2D 更适合验证确定性基础设施。2D 可在较短周期内覆盖完整流水线，3D 的四元数、惯量张量、GJK/EPA、接触流形和关节自由度会显著扩大边界条件。

### 7.2 MVP 功能清单

**必须有：**

- 固定 Tick、单线程、确定性命令缓冲；
- 选定的数值类型与完整数学库；
- 动态/静态/运动学刚体；
- 圆、胶囊、凸多边形；
- AABB 动态树或 SAP；
- SAT/解析窄相、稳定接触特征 ID；
- Sequential Impulse、warm start、摩擦、恢复；
- 距离/固定/铰链关节中的基础集合；
- 快照、恢复、重演、状态哈希、字段 diff；
- ray cast / shape cast / overlap query；
- 跨平台确定性 CI。

**可以后置：**

- 多线程、SIMD；
- CCD、睡眠；
- 动态凹体、复杂网格；
- 车辆、布娃娃、布料、流体；
- 复杂关节、电机和 breakable joint；
- GPU 物理。

GPU 物理不适合第一阶段：线程执行和规约顺序更难固定，驱动与硬件差异也扩大验证矩阵。

## 8. 分阶段实施路线

| 阶段 | 目标 | 关键交付物 | 退出标准 |
|---|---|---|---|
| M0 契约与基线 | 明确平台、规模、精度和一致性等级 | ADR、场景集、性能预算、状态 schema | 能回答“哪些平台间、多少 Tick、何种相等” |
| M1 确定性内核 | 验证数值与执行顺序 | 数学库、ID/容器、命令缓冲、RNG、哈希 | 单平台长跑逐 Tick 哈希一致 |
| M2 物理 MVP | 完整 2D 或受限 3D 流水线 | 形状、广/窄相、求解器、查询 | 典型堆叠/关节/摩擦场景稳定 |
| M3 回滚 | 任意窗口恢复与重演 | 快照环、事件提交、输入预测、diff 工具 | 回滚路径哈希等于无回滚路径 |
| M4 跨平台 | 达到 L3/L4 | Windows/Linux/macOS/ARM/WASM CI | 所有目标平台 golden hash 相同 |
| M5 优化 | 在不破坏一致性的前提下提速 | SoA、岛并行、确定性归并、profile | 达到 Tick 和追帧预算 |
| M6 产品化 | 可观测、升级和恢复 | 协议版本、资源哈希、录制包、告警 | 线上失同步可定位、可恢复 |

### M0 必须量化的指标

- 2D 还是 3D；最大动态刚体、接触和关节数；
- 目标 Tick 率、子步数、单 Tick CPU 预算；
- 最大回滚 Tick 数、最坏追帧时间、快照内存；
- 世界坐标、速度、质量、质量比和容许穿透范围；
- 目标 OS/CPU/浏览器/编译器组合；
- 是否要求版本升级前后的回放兼容；
- 是否允许服务器权威纠偏，还是必须纯 P2P。

## 9. 测试与验证体系

确定性很容易被一次优化或依赖升级破坏，因此测试体系是产品功能，不是附属工作。

### 9.1 测试层次

1. **数学单测**：边界值、舍入、溢出、负零、归一化、sqrt/trig golden vectors。
2. **几何性质测试**：A/B 交换、平移/旋转等变性、退化图形、相切、极小间隙。
3. **求解器回归**：堆叠、金字塔、关节链、摩擦斜坡、极端质量比、休眠/唤醒。
4. **重演测试**：记录输入跑 N Tick，再从头重放，逐 Tick 比哈希。
5. **回滚测试**：随机选择 Tick 恢复并重演，必须等于从未回滚的基准状态。
6. **顺序扰动测试**：随机内存地址、分配策略、线程数和任务调度，但逻辑 ID 不变。
7. **跨平台差分**：同一 replay 在所有目标平台输出逐 Tick hash stream，由 CI 比较。
8. **模糊测试**：生成合法形状、命令和高速运动；失败时保存最小 replay 包。
9. **长稳测试**：运行数十万到数百万 Tick，捕捉慢性发散、溢出与内存增长。

Box2D 会在每个 PR 跑确定性测试；Jolt 示例可在单步前保存、回绕、再执行并验证相同，这两种做法都值得纳入 CI[^box2d-sim][^jolt]。

### 9.2 CI 矩阵建议

至少包含：

- Debug / Release；
- MSVC / Clang / GCC（若属于目标工具链）；
- x86-64 / ARM64；
- Windows / Linux / macOS；
- native / WASM；
- 1、2、4、8 worker（若支持确定性并行）；
- 有/无 SIMD 的受支持组合。

每个构建产出：引擎版本、编译器版本、编译 flags、CPU feature、资源哈希、输入 replay、逐 Tick 状态哈希。首次差异要自动输出结构化状态差，而不是只报“hash mismatch”。

## 10. 现有方案对比

| 方案 | 维度 | 数值路线 | 官方声明的确定性边界 | 主要注意点 |
|---|---|---|---|---|
| Box2D 3.1+ | 2D / C17 | 受控浮点 | 64 位跨平台、跨线程；不保证 rollback determinism | 应用层仍需确定；自建完整回滚状态 |
| Jolt Physics | 3D / C++ | 受控浮点 | 开启 CMake 选项后覆盖多编译器、OS、x86/ARM/WASM | 相同源码/defines；自有 trig/sort；性能需实测 |
| Rapier | 2D/3D / Rust/WASM | 受控浮点 + 确定性数学 | 可选跨平台确定性，JS 有 deterministic 构建 | 确定性 feature 与 parallel/SIMD 冲突；初始化也须确定 |
| Photon Quantum | 2D/3D / Unity 商业方案 | Q48.16 定点 | 面向跨平台确定性与预测/回滚的完整框架 | 商业依赖；范围/精度限制；需遵循其 ECS/资产工作流 |
| 自研 | 可定制 | 定点、软件浮点或受控浮点 | 由团队定义和验证 | 成本最高，但版本、快照和协议控制力最强 |

注意：表中是各项目文档声明，不应替代本项目验收。必须用自己的碰撞规模、关节链、高速体和目标平台跑 replay/hash PoC。

### 建议的 PoC 选型流程

对 2～3 个候选方案实现同一套场景：

1. 1,000 个动态体堆叠与坍塌；
2. 长关节链和 100:1 质量比；
3. 高速弹体与 CCD；
4. 频繁创建/删除、睡眠/唤醒；
5. 每 10～30 Tick 注入一次随机深度回滚；
6. 运行 100,000 Tick，比对全部平台 hash stream；
7. 测正常 Tick P50/P95/P99、最坏回滚追帧、快照大小和恢复耗时。

只有通过这些测试，候选方案才能被称为满足本项目的“一致性”。

## 11. 常见误区与风险

### 误区 1：固定时间步就会一致

固定时间步只固定了调用次数。浮点语义、约束顺序、随机数和资源差异仍可导致分叉。

### 误区 2：IEEE 754 保证所有平台相同

基础运算标准化不代表编译器不会重排、使用 FMA，也不代表 `sin/cos/atan2` 的库实现逐位相同。必须约束工具链和数学库。

### 误区 3：定点数一定更快、更准

定点数的价值是可控语义，不是普遍性能或精度优势。整数除法、宽乘法、归一化和三角函数可能更慢，而且范围与精度必须二选一。

### 误区 4：物理引擎一致，游戏就一致

上层若用无序容器遍历实体、从系统时间取随机种子、按网络到达顺序创建对象，仍会失同步。Box2D 和 Rapier 文档都明确提醒应用层必须同样确定[^box2d-sim][^rapier-det]。

### 误区 5：PBD/XPBD 天生确定

它们只是数值求解方法，同样有迭代顺序、浮点和并行问题。选择它们应基于材料/约束需求，而不是把它们当确定性开关。

### 误区 6：只在最终帧比较结果即可

最终帧不一致只能证明有问题，无法定位首次分叉。必须逐 Tick 哈希并保留可二分的 replay。

### 误区 7：版本升级后旧回放应自然兼容

任何算法、容差、编译器或资源变更都可能改变结果。应把 simulation protocol version 放进网络握手和 replay；若要兼容旧回放，就保留旧模拟器或做离线迁移，不能依赖偶然兼容。

## 12. 最终建议

建议把项目拆成两个决策，而不是直接决定“自研一个一致性物理引擎”：

1. **先决定一致性契约**：目标平台、L1～L4 等级、回滚窗口、版本兼容和数值误差是否必须逐位为零。
2. **再决定物理实现**：成熟引擎适配还是自研；2D/3D；受控浮点、定点还是软件浮点。

在没有更多产品约束时，最稳妥的执行顺序是：

1. 用 Rapier deterministic、Jolt deterministic 或 Box2D 3.1 做两周级 PoC；
2. 先搭建 replay、全状态快照、逐 Tick 哈希和跨平台 CI；
3. 用真实场景验证 L3/L4，而不是只跑官方 demo；
4. 若成熟引擎无法提供所需的快照控制、版本稳定性或数值边界，再以 PoC 场景作为 golden suite 自研受限物理内核；
5. 自研时先单线程 2D/受限 3D，正确后再做并行、CCD、睡眠和复杂形状。

这条路线能尽早暴露真正困难的部分：通常不是 GJK 或积分公式本身，而是隐藏状态、退化几何、稳定顺序、回滚副作用和跨平台验证。
